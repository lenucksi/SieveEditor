## PR Title

```
refactor!: migrate to single-module Maven and add release automation
```

## Description

This PR modernizes the SieveEditor project infrastructure with three major improvements:

1. **Conventional Commits Enforcement** - Automated validation for consistent commit history
2. **Release Please Automation** - Complete automated release workflow configuration
3. **Maven Structure Simplification** - Refactored from multi-module to single-module project

### üéØ Motivation

The project had legacy infrastructure from when ManageSieveJ was a git submodule. After migrating to JitPack dependency (PR #26), the multi-module Maven structure was no longer needed. This PR completes the modernization by:

- Removing unnecessary complexity
- Enabling automated semantic versioning
- Improving developer experience
- Preparing for first automated release

### üìã Changes

#### 1. Conventional Commits Enforcement

**Added:**
- `.githooks/commit-msg` - Validates all commits against Conventional Commits spec
- Supports 11 commit types: `feat`, `fix`, `perf`, `security`, `deps`, `docs`, `test`, `chore`, `refactor`, `build`, `ci`
- Helpful error messages with examples when validation fails
- Automatic validation on every commit

**Enable with:**
```bash
git config core.hooksPath .githooks
```

#### 2. Claude Code Development Harness

**Added `.claude/` directory:**
- **Slash Commands**: `/build`, `/test`, `/clean`, `/package`, `/coverage`, `/verify`
- **Session Hook**: `SessionStart` - Auto-verifies environment on session start
- **Skill**: `conventional-commits` - Interactive commit message builder
- **Documentation**: Complete harness guide in `.claude/README.md`

#### 3. Developer Documentation

**Added `CLAUDE.md`** (15.5KB comprehensive guide):
- Project overview and architecture
- Complete tech stack documentation
- Development workflow with Claude Code integration
- Conventional commits requirements and examples
- Testing strategy and coverage goals
- Security considerations
- Build and packaging instructions
- Release process with Release Please
- CI/CD pipeline documentation
- Troubleshooting guide

**Updated `CONTRIBUTING.md`:**
- Added git hooks setup section (REQUIRED)
- Added Claude Code usage section
- Added `build` and `ci` commit types
- Added commit message validation guide
- Added common mistakes section

#### 4. Release Please Configuration

**Added:**
- `.release-please-manifest.json` - Tracks current version (0.0.1)
- `CHANGELOG.md` - Initial changelog in Keep a Changelog format
- Enhanced `.github/release-please-config.json`:
  - Explicit `changelog-path` specification
  - Added `build` and `ci` changelog sections
  - Removed `extra-files` (no longer needed after refactoring)

**How it works:**
1. Conventional commits pushed to `main` ‚Üí Release Please analyzes
2. Determines version bump based on commit types
3. Creates/updates Release PR with CHANGELOG
4. Merge PR ‚Üí Publishes release with multi-platform artifacts

#### 5. Maven Refactoring (BREAKING CHANGE)

**Structural changes:**
- Merged `pom.xml` (parent) + `app/pom.xml` ‚Üí single root `pom.xml`
- Changed artifactId: `sieveeditor-parent` ‚Üí `SieveEditor`
- Changed packaging: `pom` ‚Üí `jar`
- Moved `app/src/` ‚Üí `src/`
- Moved `app/README-TESTS.md` ‚Üí `README-TESTS.md`
- Deleted `app/` directory entirely

**Artifact path changes:**
- **OLD**: `app/target/SieveEditor-jar-with-dependencies.jar`
- **NEW**: `target/SieveEditor-jar-with-dependencies.jar`

**Updated workflows:**
- `.github/workflows/ci.yml` - Removed `cd app`, updated paths
- `.github/workflows/package.yml` - Removed `cd app`, updated artifact paths
- `.github/release-please-config.json` - Removed `extra-files`

**Updated documentation:**
- All build commands simplified: `mvn test` (not `cd app && mvn test`)
- Removed outdated `lib/ManageSieveJ` submodule references
- Updated Flatpak manifest JAR path

**Deleted obsolete files:**
- `update-submodules.sh` (no longer needed)
- `build.sh` (referenced old structure)
